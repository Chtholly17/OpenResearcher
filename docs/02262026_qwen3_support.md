# 02/26/2026 - Qwen3 Model Support for RL Training

Goal: Add support for Qwen3 models (starting with Qwen3-8B) as an alternative to the NemotronH-30B-A3B SFT model. Two key motivations: (1) NemotronH's SFT quality is not strong enough, (2) Qwen3 is a standard dense model without MoE/Mamba complexity, making it easier to train and debug.

## Key Differences: Nemotron vs Qwen3

| Aspect | NemotronH-30B-A3B | Qwen3-8B |
|--------|-------------------|----------|
| Architecture | 30B MoE (3B active) | 8B dense |
| Tool call format | XML (`<function=name><parameter=k>v</parameter>`) | JSON hermes (`{"name": "...", "arguments": {...}}`) |
| Answer format in SFT | `Exact Answer: ...` + `Confidence:` | Not task-specific ‚Äî teach via system prompt |
| Citation format | `„Äêcursor‚ÄÝL...„Äë` (SFT-trained) | Not used |
| `trust_remote_code` | Required | Not required |
| Parser | Custom `NemotronToolParser` | Built-in `hermes` parser |
| FSDP memory (8 GPU) | ~60 GB/GPU (need offload) | ~8 GB/GPU (no offload needed) |

## Changes Implemented

### 1. `data_utils.py` ‚Äî New Qwen3 System Prompt

Added `DEVELOPER_CONTENT_QWEN3`:

```python
DEVELOPER_CONTENT_QWEN3 = """
You are a helpful research assistant with access to web browsing tools.

You can search the web, open URLs, and find information to answer questions accurately.

The `cursor` appears in brackets before each browsing display: `[{cursor}]`.
sources=web
""".strip()
```

Key differences from `DEVELOPER_CONTENT` (Nemotron):
- No `„Äêcursor‚ÄÝL{line_start}„Äë` citation format ‚Äî Qwen3 was never SFT-trained with this syntax
- Simpler framing ("research assistant" vs "helpful and harmless assistant")

### 2. `verl_rl/preprocess_openresearcher.py` ‚Äî `--model_type` Flag

Added `ANSWER_INSTRUCTIONS_QWEN3` making `<answer>` tags the primary format:

```
When you have gathered enough information, submit your final answer using ONE of:
1. Call the submit_answer tool with your answer (wraps in <answer> tags), OR
2. Write directly in your response: <answer>your answer here</answer>
```

vs Nemotron's `ANSWER_INSTRUCTIONS` which uses `Exact Answer: [answer]` as primary.

Added `--model_type {nemotron,qwen3}` CLI argument (default: `nemotron`):
- Selects which system prompt + answer instructions to embed in the parquet
- Propagated through `make_verl_record()` and `convert_and_save()`
- Existing Nemotron data and workflow unchanged

### 3. `verl_rl/config/openresearcher_multiturn_grpo_qwen3.yaml` ‚Äî New Hydra Config

New config file (does not inherit from Nemotron config to avoid confusion):

Key settings vs Nemotron config:
- `model.path: Qwen/Qwen3-8B`
- `model.trust_remote_code: False`
- `rollout.multi_turn.format: hermes` ‚Üê critical, Nemotron uses `format: nemotron`

### 4. `verl_rl/run_grpo_training_qwen3.sh` ‚Äî New Training Script

## Architecture Notes

### Tool Call Format Difference

Nemotron outputs:
```
<tool_call>
<function=browser.search>
<parameter=query>
some query
</parameter>
</function>
</tool_call>
```

Qwen3 outputs (standard hermes/OpenAI format):
```
<tool_call>{"name": "browser.search", "arguments": {"query": "some query"}}</tool_call>
```

The `NemotronToolParser` (registered as `"nemotron"`) handles the XML format. Qwen3 uses verl's built-in hermes parser ‚Äî no custom parser needed.

### Answer Extraction

The reward function (`openresearcher_reward.py`) already supports `<answer>` tags as the primary extraction method ‚Äî no changes needed there. The Qwen3 system prompt makes this the default path, whereas Nemotron's system prompt defaults to `Exact Answer:` (since the SFT model was trained that way).

### Memory Budget (Qwen3-8B, 8√ó A100-80GB)

- Model params (bf16): 8B √ó 2B / 8 = 2 GB/GPU
- Gradients (fp32): 8B √ó 4B / 8 = 4 GB/GPU
- Optimizer (Adam): 8B √ó 8B / 8 = 8 GB/GPU
- Total FSDP: ~14 GB/GPU ‚Äî 66 GB remaining for vLLM KV cache + activations
- `gpu_memory_utilization=0.6` leaves ~32 GB for FSDP during rollout sleep
